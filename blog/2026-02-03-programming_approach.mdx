### <B2>嵌入式 API 设计哲学：契约式设计 vs 防御性编程</B2>

本文介绍契约式设计与防御性编程。

<!-- truncate -->

📝 **博客节选：契约式设计 vs 防御性编程 —— 嵌入式开发中的 API 设计哲学**

在学习环形缓冲区实现时，我看到很多教程（包括权威资料）用 `assert` 校验参数，理由是：“避免代码中充斥条件判断，用 *Design by Contract*（契约式设计）明确 API 使用前提。”

这确实有道理：若调用者严格遵守约定（如不传 `NULL`、不传非法值），内部无需冗余检查，逻辑更简洁、性能更高。

但现实世界并不完美：
- 调用者可能是未来的你，加班手抖传了 `0`；
- 可能是新人，还没读完文档；
- 甚至是自动生成的代码，存在边界 case。

此时，`assert` 触发后静默死机（停在 `BKPT 0`）对用户毫无帮助——**无提示、无法恢复、难定位**，尤其在无调试器的现场设备上。

因此，在**面向用户的公共 API** 中，我更倾向采用**防御性编程**：
- 显式校验输入参数；
- 返回明确错误码（如 `CBUF_ERR_INVALID_ARG`）；
- 允许调用者重试、降级或记录日志。

这不仅提升健壮性，也降低维护成本。

✅ **我的实践建议：分层使用，各取所长**
- **对外 API**（如 `circular_buf_init`）→ 返回错误码，友好容错；  
- **内部函数**（仅库自身调用）→ 用 `assert` 快速暴露逻辑错误；  
- **开发阶段** → 启用 `assert` 辅助调试；  
- **发布版本** → 通过 `#define NDEBUG` 移除 `assert`，但保留错误处理路径。

> 契约式设计告诉我们“什么是对的”，防御性编程则帮我们应对“什么是错的”。  
> 在资源受限、无人值守的嵌入式系统中，后者往往更为关键。

最终，好的 API 不仅要高效，更要**宽容、可预测、可恢复**——  
因为真正的可靠性，不在于“永不犯错”，而在于“犯错后仍能优雅应对”。


### `assert` 使用指南：从开发到部署的完整实践

`assert` 是 C 语言中用于**运行时契约检查**的调试工具，正确使用能极大提升开发效率，但需注意其局限性。

---

#### ✅ 正确用法

1. **包含头文件**
```c
#include <assert.h>
assert(expression);  // expression 为假（0）时触发失败
```

### 典型场景
检查函数参数合法性（如指针非空、`size > 0`）
验证内部状态一致性（如缓冲区满/空标志逻辑）
确保前置条件成立（如“调用前必须初始化”）
🔧 调试技巧（Keil + Cortex-M）
你当前的方法——开启 Debug Session 并设断点——是标准做法。补充建议：
无需手动设断点：assert 失败时会自动停在 BKPT 0 指令处；
查看调用栈：在 Keil 的 Call Stack + Locals 窗口，可直接看到：
哪一行触发了断言（__LINE__）
失败的表达式（如 "buffer && size"）
调用路径（如 main → circular_buf_init）
单步进入：若未停住，说明 assert 被禁用（见下文）
💡 提示：确保已实现 __aeabi_assert（含 BKPT 0），否则可能静默 HardFault。
⚠️ 常见陷阱
表格
问题	原因	解决
assert 不生效	定义了 NDEBUG	检查是否在 `#include <assert.h>` 前或 Keil 工程中定义了 NDEBUG
链接报错 __aeabi_assert 未定义	未提供底层实现	在任意 .c 文件中实现 __aeabi_assert（含 BKPT 0 + 死循环）
程序死机但无提示	无调试器且未做产品级处理	发布版本应移除 assert 或替换为日志/重启等行为
📦 开发 vs 发布策略
表格
阶段	建议
开发/测试	启用 assert，配合调试器快速定位问题
发布/产品	定义 NDEBUG 移除 assert，但关键 API 应保留错误码校验
📌 黄金法则：
assert 用于捕获本不该发生的程序逻辑错误，
而用户可能犯的错误（如传参非法）应通过返回值处理。
✅ 最佳实践总结
仅用于内部逻辑自检，不用于公共 API 的输入校验；
永远假设最终产品没有调试器，关键路径需有 fallback；
结合 #ifdef DEBUG 或自定义宏（如 CBUF_ASSERT）实现灵活开关；
在嵌入式系统中，优先考虑静态内存 + 错误码，而非依赖 assert 保证安全。
assert 是开发者的“哨兵”，不是产品的“保险”。

assert相关[解释](https://www.runoob.com/w3cnote/c-assert.html)

> 如果函数已有错误返回机制，就用它来承载参数校验；
如果没有，就靠契约（assert）保证输入合法
