## 串口发送方法

| 需求 | 推荐方案 |
|------|--------|
| 调试打印（`printf`） | 阻塞轮询（简单） |
| 常规通信（Modbus/AT/自定义协议） | ✅ 中断 + 环形缓冲区 |
| 高速连续流（>100KB/s） | DMA |

### 阻塞轮询（简单）
配置，以串口3为例：
```
void Usart3_Init(u32 baud)
{
	RCC_APB1PeriphClockCmd(RCC_APB1Periph_USART3,ENABLE);//开启USART3时钟
	RCC_APB2PeriphClockCmd(RCC_APB2Periph_GPIOB,ENABLE);//开启GPIOB时钟，因为需要用到PB10与PB11
	/*
	下面是把PB10配置为复用推挽输出，供USART3的TX使用
	*/
	GPIO_InitTypeDef GPIO_InitStructure;
	GPIO_InitStructure.GPIO_Mode=GPIO_Mode_AF_PP;
	GPIO_InitStructure.GPIO_Pin=GPIO_Pin_10;
	GPIO_InitStructure.GPIO_Speed=GPIO_Speed_50MHz;
	GPIO_Init(GPIOB,&GPIO_InitStructure);
	
	GPIO_InitStructure.GPIO_Mode=GPIO_Mode_IPU;//对于输入可以选用浮空输入或上拉输入模式
	GPIO_InitStructure.GPIO_Pin=GPIO_Pin_11;
	GPIO_InitStructure.GPIO_Speed=GPIO_Speed_50MHz;
	GPIO_Init(GPIOB,&GPIO_InitStructure);
	/*
	下面串口配置情况：
	9600波特率，8位字长，无校验，1位停止位，无流控，
	*/
	USART_InitTypeDef USART_InitStructure;
	USART_InitStructure.USART_BaudRate=baud;//设置波特率
	USART_InitStructure.USART_HardwareFlowControl=USART_HardwareFlowControl_None;
	USART_InitStructure.USART_Mode=USART_Mode_Tx|USART_Mode_Rx;//同时开启发送与接收
	USART_InitStructure.USART_Parity=USART_Parity_No;
	USART_InitStructure.USART_StopBits=USART_StopBits_1;
	USART_InitStructure.USART_WordLength=USART_WordLength_8b;
	USART_Init(USART3,&USART_InitStructure);
	USART_ITConfig(USART3,USART_IT_RXNE,ENABLE);//此为开启RXNE标志位到NVIC的输出
	/*
	一旦RXNE标志位一旦置1了，就会向NVIC申请中断，之后可以在中断函数里接收数据
	*/
	NVIC_PriorityGroupConfig(NVIC_PriorityGroup_2);
	
	NVIC_InitTypeDef NVIC_InitStructure;
	NVIC_InitStructure.NVIC_IRQChannel=USART3_IRQn;
	NVIC_InitStructure.NVIC_IRQChannelCmd=ENABLE;
	NVIC_InitStructure.NVIC_IRQChannelPreemptionPriority=1;
	NVIC_InitStructure.NVIC_IRQChannelSubPriority=1;
	NVIC_Init(&NVIC_InitStructure);
	//使能串口
	USART_Cmd(USART3,ENABLE);
	
}
```
串口发送可能有多个函数，比如`Serial_SendArray`、`Serial_SendString`、`Serial_Printf`等等，但最终都会回归到这这个最本质的函数：
```c
void Serial_SendByte(USART_TypeDef *USARTx,uint8_t Byte)
{
	USART_SendData(USARTx,Byte);//此函数将Byte变量写入到TDR
	while(USART_GetFlagStatus(USARTx,USART_FLAG_TXE)==RESET);//等待TDR的数据转移到移位寄存器
}
```

#### 🧱 二、为什么这样做？—— 背后的工程权衡
1. 发送用轮询：因为“简单且够用”
* 发送是主动行为：主程序决定何时发、发什么
* 低频场景下无感知：比如 Serial_Printf("Hello\n") 发 6 字节，在 9600bps 下仅阻塞约 6ms
* 无需复杂状态管理：不像接收那样要应对“未知何时来数据”的问题
* 调试友好：Serial_Printf 卡住 = 程序卡住，容易定位问题
> 💡 对于 调试输出、启动日志、低频命令响应，轮询完全够用。

#### ⚠️ 四、但它的致命缺陷是什么？
虽然简单，但在产品级应用中会暴露严重问题：
#### ❌ 1. 发送阻塞主程序
* 如果你在 while(1) 中频繁打印，系统无法及时响应按键、传感器等
* 在高波特率下（如 115200），影响较小；但在 9600 下，发 100 字节 ≈ 100ms 阻塞！

#### ❌ 3. 无法在中断中安全调用发送函数
如果你在其他中断（如定时器）中调用 Serial_SendString()，
而此时 UART 正在发送，就会 死等 TXE → 中断嵌套卡死

<summary>阻塞发送中断死锁分析</summary>

#### 假设系统中有两个中断：
1. UART 发送中（比如主循环刚调用了 `Serial_SendString`）
2. 定时器中断（每 1ms 触发一次，用于心跳或日志）
```c
// 主循环
int main(void) {
    Serial_Init();
    Timer_Init(); // 启动 1ms 定时器中断
    
    while (1) {
        Serial_SendString("Main loop running...\r\n"); // 阻塞发送
        Delay_ms(100);
    }
}

// 定时器中断（高优先级）
void TIM2_IRQHandler(void) {
    if (TIM_GetITStatus(TIM2, TIM_IT_Update)) {
        // 每 1ms 打印一次心跳（看似合理的需求）
        Serial_SendString("Heartbeat\r\n"); // ⚠️ 危险！
        TIM_ClearITPendingBit(TIM2, TIM_IT_Update);
    }
}
```

🔥 **死锁发生过程**

**步骤 1：主循环开始发送**  
- `Serial_SendString("Main loop running...\r\n")` 被调用（20+字节）  
- 逐字节发送，当前发送 `'M'` 字节  
- UART 硬件忙碌，`TXE = 0`

**步骤 2：高优先级中断抢占**  
- 定时器中断触发（优先级 > 主循环）  
- CPU 暂停主循环的 `while(USART_FLAG_TXE == RESET)`  
- 跳转到 `TIM2_IRQHandler()`

**步骤 3：中断中再次发送**  
- 中断服务程序调用 `Serial_SendString("Heartbeat\r\n")`（10+字节）  
- 执行 `Serial_SendByte('H')` 时等待 `TXE` 标志  
- **死等开始**：UART 仍在处理主循环的剩余数据，`TXE` 保持为 `0`

**步骤 4：死锁形成**  
- **主循环**：等待剩余 18+ 字节发送完成（被中断抢占无法继续）  
- **中断**：等待 `TXE=1` 才能发送（主循环数据未完成，`TXE` 不会变 `1`）  
- **结果**：互相等待，系统永久卡死

🚫 **核心问题**  
UART 硬件资源竞争：主循环的多字节发送过程被中断打断，中断中又尝试访问同一 UART 资源，造成不可解的死锁。

---


| 层级         | 状态                                      |
|--------------|-------------------------------------------|
| **主循环**   | 卡在 `while(!TXE)`，等待 UART 空闲        |
| **TIM2 中断**| 卡在 `while(!TXE)`，也等待 UART 空闲      |
| **UART 硬件**| 正在发送 `'M'`，完成后会置 `TXE=1`        |

但问题在于：
- 只有主循环能处理完当前发送（因为它在发 `"Main loop..."` 的后续字节）
- 但主循环被 TIM2 中断抢占了
- TIM2 中断又在等同一个 UART 资源  
→ **资源竞争 + 优先级反转 = 死锁**

#### 如何安全使用阻塞式串口发送？


虽然 `Serial_SendByte()` 是阻塞轮询实现，但只要遵守以下 **三条简单规则**，就能在大多数场景中安全使用：


#### ✅ 安全使用原则

**1. 仅在主循环（或低优先级任务）中调用**
- ❌ 禁止在任何中断服务程序（ISR）中调用 `Serial_SendXXX`
- ✅ 只在 `main()` 的 `while(1)` 或 FreeRTOS 任务中使用

> **原因**：避免中断嵌套死锁（见死锁分析）


**2. 控制发送频率和数据量**
- 单次发送 ≤ 20 字节（9600bps 下 ≈ 20ms 阻塞）
- 发送间隔 ≥ 50ms（给人“不卡”的感觉）
- 示例安全用法：
  ```c
  while (1) {
      Debug_Printf("Sensor: %d\n", value); // OK：低频
      Delay_ms(100);
  }
  ```
> 原因：避免长时间阻塞影响实时响应

**3. 绝不用于协议通信或高频日志**
* ❌ 不要用它发送 AT 指令、Modbus 帧等协议数据
* ❌ 不要在高速采样循环中打印（如 for(i=0;`i<1000`;i++) printf(...)）
* ✅ 仅用于：启动信息、人工交互、偶发调试
* 原因：协议通信需要可靠、非阻塞的发送机制


#### 📊 五、适用场景 vs 不适用场景
| ✅ 适用场景 | ❌ 不适用场景 |
|-----------|--------------|
| 调试打印（`printf`） | 协议通信（AT/Modbus/NMEA） |
| 启动阶段状态输出 | 高频数据上报（>10Hz） |
| 交互式命令行（人工输入） | 多任务/RTOS 环境 |
| 学习 UART 基础原理 | 产品级稳定通信 |

### 中断 + 环形缓冲区

串口中断 + 环形缓冲区是如何配合实现“非阻塞发送”的？
🧺 比喻：邮局寄信（你 = 主程序，邮局 = 串口，邮箱 = 环形缓冲区）
想象你要寄很多封信（比如 100 封），但你很忙，不能站在邮局等每封信寄出去。
❌ 老方法（轮询/阻塞发送）：
你写一封信 → 跑到邮局 → 亲手交给邮递员 → 等他放进邮筒 → 才能回去写第二封
问题：你被“卡”在邮局，啥也干不了！
✅ 新方法（中断 + 环形缓冲区）：
你家门口有个邮箱（环形缓冲区）
你只管把信扔进邮箱（调用 Serial_SendByte()），立刻回去工作
邮递员（串口中断） 定期来检查邮箱：
如果有信 → 拿走一封，放进邮筒（通过 TX 引脚发出去）
发完一封 → 马上回来问：“邮箱还有信吗？”
有就继续发，没就回家休息
你完全不用等，效率极高！
🔧 对应到 STM32 代码
1️⃣ 环形缓冲区（你的邮箱）
```c
static uint8_t tx_buffer[128];      // 邮箱容量：128 封信
static cbuf_handle_t tx_cbuf;       // 邮箱管理器（记录哪空哪满）
static volatile bool tx_sending = false; // 邮递员是否在工作？

```
2️⃣ 你扔信（主程序调用发送函数）

```c
int Serial_SendByte(USART_TypeDef *USARTx, uint8_t Byte)
{
    __disable_irq(); // 关门！防止邮递员和你同时操作邮箱（原子性）
    int ret = circular_buf_put(tx_cbuf, Byte); // 把信塞进邮箱
    __enable_irq();

    if (ret == 0) { // 塞成功了！
        if (!tx_sending) { // 如果邮递员在家睡觉
            tx_sending = true;
            USART_ITConfig(USARTx, USART_IT_TXE, ENABLE); // 叫醒他！
        }
    }
    return ret; // 成功=0，失败≠0（邮箱满了）
}
```
✅ 关键：这个函数立即返回！你不用等信寄出。
3️⃣ 邮递员工作（串口中断服务程序）
```c
void USART3_IRQHandler(void)
{
    // ... RX 部分先不管 ...

    // TX 部分：邮递员被叫醒了！
    if (USART_GetITStatus(USART3, USART_IT_TXE) != RESET)
    {
        uint8_t data;
        if (circular_buf_get(tx_cbuf, &data) == 0) {
            // 邮箱里有信！拿出来
            USART_SendData(USART3, data); // 放进邮筒（发出去）
        } else {
            // 邮箱空了！
            tx_sending = false;
            USART_ITConfig(USART3, USART_IT_TXE, DISABLE); // 回家睡觉
        }
        USART_ClearITPendingBit(USART3, USART_IT_TXE);
    }
}
```
✅ 关键：
每次中断只发 1 个字节（1 封信）
发完立刻检查邮箱是否还有信
没有了就关中断（省电+减少打扰）

✅ 这样设计的好处
表格
优点	说明
非阻塞	主程序调用发送后立刻继续干活
高吞吐	中断自动连续发送，不浪费 CPU
防溢出	缓冲区满时自动丢弃或报错，不崩溃

⚠️ 注意事项（你遇到的问题根源）
邮箱太小（缓冲区小）：信太多会塞不下 → 后面的信丢掉
你扔信太快：邮递员还没清空邮箱，你就又塞满 → 新信进不来
但系统不会死：只是丢信（数据），程序继续跑
这就是为什么你看到 big_msg 被截断 —— 不是中断坏了，是邮箱满了！
💡 总结一句话
环形缓冲区是“临时仓库”，串口中断是“搬运工”。
主程序只管往仓库放货，搬运工自动把货发出去。
两者配合，实现高效、不卡顿的串口发送。

不过，目前我的环形缓冲区是能和串口配合并正常工作的，这倒是让我欣慰。

#### 缓冲区满/溢出理解

假设我设定的缓冲区大小是128字节，那么用slot版本的话，环形缓冲区容量就有127字节可以装。现在我构造一个长字符big_msg，长度为128字节，看似超了一字节，会溢出，实则不会，因为压入和发送是异步进行的。再来个例子，如果我构造一个短字符short_msg，长度为80字节，看似远低于127，应该绝对不会溢出？实则还是可能会，因为如果原来的缓冲区已经有一部分数据，但这时候由于某些原因，中断没来得及处理，也就是数据发送出去的速度过慢，那么即使是short_msg，也会被截断。

所以，决定数据是否会被截断，不是缓冲区大小决定，而是数据生产与数据消费速度共同决定的。如果消费速度足够快，远快于生产，那么即使数据超过缓冲区容量，也不会截断。最好的办法是，尽量保证数据生产与数据消费速度一致。


✅ “决定数据是否被截断的，不是缓冲区大小本身，而是生产速度与消费速度的动态关系。”
✅ “即使消息很短，若缓冲区已满且消费停滞，也会被截断。”
✅ “即使消息略超容量，若消费及时，也可能完整入队。”

## 环形缓冲区的真相：为什么“127字节容量”并不意味着“128字节必丢”

在嵌入式开发中，环形缓冲区（Circular Buffer）是串口、DMA、任务通信等场景的核心组件。我们常听说：“缓冲区大小为 N，实际可用容量为 N-1”。但真实世界远比这复杂——**是否溢出，不只看容量，更看“时机”**。

本文通过一个真实的突发数据测试案例，揭示环形缓冲区在异步系统中的动态行为本质。

---

## 一、经典误解：静态容量决定一切？

假设我们定义：
```c
#define TX_BUF_SIZE 128
// 实际可用容量 = 127 字节（牺牲1 slot区分空/满）
```
很多人认为：  
- 发送 ≤127 字节 → 一定成功  
- 发送 ≥130 字节 → 一定截断  

但实测发现：  
- **129 字节的消息完整发送成功**  
- **80 字节的短消息反而被丢弃**  

为什么？因为**缓冲区不是静态水桶，而是动态流水线**。

## 二、关键机制：生产者与消费者的异步博弈

1. **生产者（主循环）**  
   - 调用 `Serial_SendString()` 将数据压入缓冲区  
   - 速度极快（CPU 微秒级操作）

2. **消费者（USART 中断）**  
   - 逐字节从缓冲区取出数据发送  
   - 速度由波特率决定（如 9600 bps ≈ 960 B/s）

3. **核心结论**  
   > **是否溢出，取决于“写入过程中能否动态腾出空间”，而非初始剩余容量。**

#### 三、实验验证：+2 成功，+3 失败
我们构造接近极限的消息：   
```c
int prefix_len = strlen(">>> Burst test: max-capacity data:["); // 
//- 1 是因为少了一个slot，-3 是为后面的]\r\n留位置
int fill_len = TX_BUF_SIZE - prefix_len - 1 - 3 + OFFSET; // OFFSET = +2 or +3
// 总长度 = prefix_len  + fill_len + 3(]\r\n) = 130
```
| OFFSET | 总长度 | 是否完整换行 | 原因 |
|--------|--------|--------------|------|
| `+2`   | 129 B  | ✅ 是        | 写入期间中断腾出 ≥2 字节 |
| `+3`   | 130 B  | ❌ 否（`\n` 丢失） | 缓冲区始终无法容纳第130字节 |

> 💡 129 > 127 却成功，证明“动态腾空”真实存在！

同时观察到：在 burst 发送后，下一条 `"Raw array: ..."` 被丢弃——因为此时缓冲区仍满，**短消息也会失败。**

#### 四、正确理解溢出条件
#### ❌ 错误认知
> “只要消息长度 ≤ 127，就安全。”
#### ✅ 正确认知
> **消息能否完整入队，取决于以下因素的实时组合**：
> * 缓冲区当前剩余空间
> * 消息总长度
> * USART 中断响应延迟
> * CPU 是否被高优先级任务占用
> * `SendString` 的实现方式（逐字节 vs 批量检查）

**示例场景**
| 场景 | 结果 |
|------|------|
| 缓冲区空 + 发 129B | ✅ 可能成功（中断及时腾空） |
| 缓冲区剩 50B + 发 80B | ❌ 可能截断（即使 80 < 127） |
| 高负载下连续发 10 次 127B | ⚠️ 后续批次可能丢数据（平均速率超限） |
> 之所以说可能是因为生产和消费是伪同步的，如果一定要说必然截断，那就是目前缓冲区有50B，而且发送被意外中止了，那么这个时候压入 80B 便会必然截断。
#### 五、工程最佳实践
#### 1. 发送函数必须逐字节入队
```c
// ✅ 推荐：边发边试
while (*str) {
    if (!circular_buf_put(cbuf, *str)) break;
    str++;
}

// ❌ 避免：一次性检查
if (strlen(str) <= circular_buf_free(cbuf)) {
    // 批量复制 → 无法利用动态腾空
}
```
2. **缓冲区设计原则**  
   - 大小 ≥ 最大突发数据量 −（突发持续时间 × 波特率吞吐）  
   - 例如：最大突发 200 B，波特率 115200（≈11.5 KB/s），突发持续 10 ms → 需缓冲区 ≥ 200 − 115 = **85 B**

3. **速率控制策略**  
   - 不追求“瞬时生产 = 消费”，而要保证 **长期平均生产速率 ≤ 消费速率**  
   - 对关键数据增加重传或软件流控（如 XON/XOFF）

4. **调试技巧**  
   - 用 OLED / 调试串口显示 `strlen()` 和缓冲区状态  
   - 主动测试 `+2` / `+3` 等边界情况，验证系统鲁棒性

#### 六、结语

环形缓冲区不是“固定容量的盒子”，而是**连接生产与消费的动态桥梁**。  
它的价值不在于“存多少”，而在于“**平滑突发、解耦速率**”。

> **真正的稳健，来自于对异步本质的理解，而非对静态数字的迷信。**

下次当你看到“缓冲区大小 128”，请记住：  
**在中断的节奏中，129 也可能安然通过。**

## 环形缓冲区调试经历总结

## 问题现象
- 测试 `test_max_instances()` 时程序“卡住”，串口输出不完整。
- 原因：断言触发 `__asm { BKPT 0 }`，CPU 暂停等待调试器，未进入后续死循环。

## 根本原因
- 开发阶段启用了 `CBUF_ENABLE_ASSERTIONS`，断言失败时执行 `BKPT`。
- 调试器未继续运行，导致程序看似“死机”。
- 同时发现部分函数（如 `circular_buf_put`）包含危险检查：`if (!me->buffer)`。

## 关键认知
- `me && me->buffer` 在断言中是安全的（短路求值）。
- 但运行时 `if (!me->buffer)` 在 `me` 为野指针时会非法访问内存 → HardFault。
- 合法句柄由 `init` 返回，其 `buffer` 永不为 NULL；野指针无法安全检测。

## 解决方法
1. **测试前关闭断言**：避免 `BKPT` 阻塞自动化测试。
2. **移除多余检查**：删除 `!me->buffer` 等危险判断，仅保留 `!me`。
3. **明确 API 契约**：
   - 允许传 `NULL`（返回错误）；
   - 非 `NULL` 必须是 `init` 返回的有效句柄。
4. **依赖断言暴露 bug**：开发阶段快速捕获野指针或逻辑错误。

## 反思
- 断言用于“快速失败”，不是运行时防护。
- C 语言无法安全验证野指针，应靠设计契约 + 调试手段保证正确性。
- 边界测试（如传野指针）虽不现实，但有助于发现设计缺陷。
